The architecture

A CQRS microservice sprawl → one modern monolith

Eleven repos glued by a bespoke event bus → one modern monolith.

Today - event-sourced sprawl

  • bc-esBespoke event store + CQRS - glue nobody else maintains
  • 3× ClojureScript LambdasM365 · ConnectWise · cron · + bc-notifications (Clojure)
  • beecastle-webSeparate React frontend (220K LOC)
  • 11 repos · 3 languages+ bc-api, Keycloak, admin, loaders - an event bus between them all
→

After - a modern monolith

  • beecastle-serverOne Next.js app: UI + API + workers
  • Postgres 18 + DrizzleDirect writes + audit_log - no event store
  • Lambda + SQS - only at the edgeDurable webhook ingest · BullMQ in-repo
  • 1 repo · 1 runtime · 1 CIStrict TypeScript · Better Auth replaces Keycloak

Built on the most popular stack in software

% of pro developers using · Stack Overflow Developer Survey 2025 (n≈49k)

*Legacy core. ClojureScript <1% - barely registers.

Bus-factor & speed, by construction

The rebuild's stack is the industry default - the largest hiring pool on earth, so it's safe if I'm not here. It's also what LLM coding agents are most fluent in (the most training data) - precisely why this build moves this fast. The legacy Clojure core sits at ~2%: niche to hire for, slow for AI to touch.

CQRS + event-sourcing solved a problem BeeCastle doesn't have - a monolith on the default stack does.

04 / 13

The real problem

The rewrite removes the pain customers already feel.

BeeCastle's biggest problem is not "old tech". It is that core product behaviours are fragile, expensive, and too dependent on manual support. The rewrite replaces hidden failure modes with observable, self-serve, auditable systems.

✓ Rewrite Product-shaped Postgres model.

✓ Rewrite Deterministic revenue pipeline.

AI Intelligence
